Materia Integración de Sistemas
Herramienta Docker Compose
Stack React · Express · PostgreSQL

0 Objetivo de la Práctica

En esta práctica aplicarás los conceptos de orquestación de contenedores con Docker Compose levantando una aplicación real compuesta por tres servicios independientes que se comunican entre sí a través de una red virtual creada por Docker.

🎯
¿Qué aprenderás?
Escribir un docker-compose.yml completo · Definir servicios, redes y volúmenes · Gestionar dependencias entre contenedores · Observar métricas en tiempo real · Interactuar con la BD desde una API REST.

3 Análisis del docker-compose.yml

Abre el archivo docker-compose.yml y analiza cada sección.

Servicio PostgreSQL

postgres:
  image: postgres:15-alpine       # Imagen oficial, versión ligera
  container_name: lab_postgres
  environment:                    # Variables de entorno (credenciales)
    POSTGRES_USER:     admin
    POSTGRES_PASSWORD: secret123
    POSTGRES_DB:       dockerlab
  volumes:
    - postgres_data:/var/lib/postgresql/data  # Persistencia
    - ./postgres/init.sql:/docker-entrypoint-initdb.d/init.sql
  healthcheck:                    # Verifica que la BD esté lista
    test: ["CMD-SHELL", "pg_isready -U admin -d dockerlab"]
    interval: 10s
    retries: 5

Servicio Backend

backend:
  build:
    context: ./backend            # Carpeta con el Dockerfile
    dockerfile: Dockerfile
  environment:
    DB_HOST: postgres              # ← Nombre del servicio, no IP
    DB_PORT: 5432
    DB_USER: admin
    DB_PASSWORD: secret123
  depends_on:
    postgres:
      condition: service_healthy  # Espera al healthcheck
  networks:
    - lab_network
💡
DNS interno de Docker Compose
Docker Compose crea una red virtual donde cada servicio puede ser accedido por su nombre (ej: postgres, backend). No necesitas conocer las IPs — Docker resuelve los nombres automáticamente dentro de la red.

Red y volúmenes

volumes:
  postgres_data:                   # Volumen named — persiste entre reinicios

networks:
  lab_network:
    driver: bridge               # Tipo de red (bridge = aislada)

4 Ejecutar el Proyecto

  1. Abrir una terminal en la carpeta del proyecto Navega hasta el directorio docker-compose-lab/ 
    $ cd docker-compose-lab
  2. Construir las imágenes y levantar los servicios Este comando descarga la imagen de PostgreSQL y construye las imágenes de backend y frontend. 
    $ docker compose up --build

    Observa cómo el output muestra los tres servicios iniciándose. El backend esperará al healthcheck de PostgreSQL antes de conectarse.

  3. Verificar que los servicios están corriendo Abre otra terminal y ejecuta: 
    $ docker compose ps
    Deberías ver los tres contenedores con estado running o healthy.
  4. Abrir la aplicación en el navegador
    ServicioURLDescripción
    Frontendhttp://localhost:3000Dashboard con métricas en tiempo real
    Backend APIhttp://localhost:4000/healthHealth check del backend
    Métricashttp://localhost:4000/api/metricsJSON con métricas del contenedor

5 Exploración del Dashboard

La aplicación React muestra tres paneles. Explóralos todos:

📊 Panel de Métricas (pestaña "Métricas")

Muestra en tiempo real (actualización cada 3 segundos):

  • CPU Load — carga del CPU del host
  • Mem Host % — porcentaje de memoria usada
  • Heap Node.js — memoria usada por el proceso Express
  • Estado de PostgreSQL — versión, latencia y tamaño de la BD
  • Sparklines — historial gráfico de CPU y memoria

✅ Panel de Tareas (pestaña "Tareas")

Gestiona tareas almacenadas en PostgreSQL. Puedes:

  • Agregar nuevas tareas (se guardan en la BD)
  • Marcar tareas como completadas
  • Eliminar tareas

📋 Panel de Logs (pestaña "Logs")

Muestra los eventos registrados en la tabla container_logs de PostgreSQL.

7 Ejercicios Prácticos

🔬
Instrucciones generales
Realiza cada ejercicio en orden. Documenta tus observaciones con capturas de pantalla. Al final entrega evidencia de cada ejercicio completado.

Ejercicio 1 — Levantamiento y verificación inicial

  1. Ejecuta docker compose up --build y observa el orden de inicio de los servicios. ¿Por qué postgres inicia primero?
  2. Ejecuta docker compose ps y anota el estado de cada contenedor.
  3. Abre http://localhost:3000 y toma una captura de pantalla del dashboard.
  4. Responde: ¿Qué diferencia hay entre image: y build: en el docker-compose.yml?

Ejercicio 2 — Redes y comunicación

  1. Entra al contenedor del backend:
    docker compose exec backend sh
  2. Dentro del contenedor, intenta hacer ping al servicio de base de datos:
    ping postgres
    ¿Qué observas? ¿Por qué funciona usando el nombre postgres?
  3. Sal del contenedor con exit.
  4. Responde: ¿Qué es el driver bridge en Docker Compose?

Ejercicio 3 — Persistencia de datos

  1. Abre la pestaña "Tareas" y agrega 3 tareas nuevas con tus nombres.
  2. Detén los servicios con docker compose down (sin -v).
  3. Vuelve a levantar con docker compose up. ¿Siguen tus tareas? ¿Por qué?
  4. Ahora ejecuta docker compose down -v. Levanta de nuevo. ¿Qué pasó con las tareas?
  5. Responde: ¿Cuál es la diferencia entre un volumen named y un bind mount?

Ejercicio 4 — Escalabilidad y variables de entorno

  1. Modifica el docker-compose.yml para agregar la variable de entorno APP_ENV: production al servicio backend.
  2. Reconstruye con docker compose up --build.
  3. Verifica que la variable existe dentro del contenedor:
    docker compose exec backend sh -c "echo $APP_ENV"
  4. Responde: ¿Por qué es mejor pasar credenciales como variables de entorno en lugar de hardcodearlas en el código?

Ejercicio 5 — Consultas directas a PostgreSQL

  1. Conéctate a PostgreSQL:
    docker compose exec postgres psql -U admin -d dockerlab
  2. Ejecuta las siguientes consultas: 
    -- Ver tablas
\dt

-- Ver todas las tareas
SELECT * FROM tasks;

-- Insertar una tarea directamente
INSERT INTO tasks (title, status) VALUES ('Tarea desde SQL', 'pending');

-- Ver logs de actividad
SELECT * FROM container_logs ORDER BY logged_at DESC;

-- Salir
\q
  3. Recarga el dashboard. ¿Aparece la tarea que insertaste directamente en SQL?

8 Preguntas de Reflexión

Responde en tu informe las siguientes preguntas:

  1. ¿Cuál es la diferencia entre el modelo declarativo de Docker Compose y el imperativo de Docker puro?
  2. ¿Por qué se usa depends_on con condition: service_healthy y no solo depends_on: postgres?
  3. Si necesitas escalar el servicio backend a 3 instancias, ¿qué cambio harías en el compose?
  4. ¿Cómo garantiza Docker Compose que los contenedores de la misma aplicación se comuniquen sin exponer puertos innecesarios al exterior?
  5. ¿En qué se diferencia Docker Compose de Kubernetes para entornos empresariales?
